# 세움비즈 업체 포털 기획 문서

이 문서는 세움비즈의 구현 방향을 정리한 포털 설계 초안입니다.
현재 단계에서는 코드, 화면, DB, API, 로그인 기능을 구현하지 않습니다.

## 1. 전체 서비스 개념

세움비즈는 랜딩페이지가 아니라 업체 전용 운영 포털입니다.

업체는 로그인 후 다음 업무를 처리합니다.

- 상품권 매입 신청
- 매입 신청 처리 상태 확인
- 업체 잔액 내역 확인
- 업체 잔액 출금 신청
- 내 정보 관리

운영자는 관리자 화면에서 다음 업무를 처리합니다.

- 업체 및 담당자 관리
- 매입 신청 검수
- 승인/반려/취소 처리
- 업체 잔액 반영
- 업체 잔액 출금 요청 확인
- 카카오톡 계좌 확인 후 수동 이체
- 지급승인 및 잔액 차감

## 2. 화면 구조 원칙

- 좌측 메뉴 기반
- PC 우선
- 모바일 대응
- 밝은 금융 B2B 톤
- 화이트/네이비/블루 계열
- 업무용 테이블, 필터, 상태 배지 중심
- 과한 광고형 랜딩 구성 지양
- 로그인 후 업무 처리 흐름을 우선

## 3. 좌측 메뉴 구조

업체 포털 메뉴:

1. 매입 신청
2. 매입 내역
3. 업체 잔액 내역
4. 업체 잔액 출금
5. 내 정보 관리

각 메뉴의 기본 역할:

| 메뉴 | 역할 |
| --- | --- |
| 매입 신청 | 상품권 종류 선택, 핀번호 입력, 이미지 업로드, 추출 결과 확인 후 최종 등록 |
| 매입 내역 | 신청 건별 승인대기, 승인완료, 반려, 취소 상태 확인 |
| 업체 잔액 내역 | 업체 잔액 반영, 차감, 조정 로그 확인 |
| 업체 잔액 출금 | 출금 금액 신청 및 처리 상태 확인 |
| 내 정보 관리 | 담당자 정보와 연락처 확인 및 수정 |

## 4. 매입 신청 화면 구조

매입 신청 화면은 상품권을 빠르게 등록하되, 자동 추출 결과를 바로 저장하지 않고 업체 확인 단계를 반드시 둡니다.

화면 구성:

- 상품권 종류 선택
- 상품권 안내
- 핀번호 전체 붙여넣기
- 핀번호 직접 입력
- 상품권 이미지 업로드
- 이미지에서 핀번호 자동 추출
- 추출 결과 미리보기
- 업체 확인/수정 영역
- 최종 등록 버튼

신청 흐름:

1. 업체가 상품권 종류를 선택합니다.
2. 상품권별 안내와 입력 규칙을 확인합니다.
3. 업체가 핀번호를 직접 입력하거나 전체 붙여넣기를 합니다.
4. 또는 상품권 이미지를 업로드합니다.
5. 시스템이 이미지에서 핀번호 후보를 추출합니다.
6. 추출 결과를 미리보기로 보여줍니다.
7. 업체가 추출 결과를 확인하고 수정합니다.
8. 업체가 최종 등록 버튼을 눌러 매입 신청을 생성합니다.
9. 상태는 승인대기가 됩니다.
10. 관리자 검수 후 승인완료 또는 반려로 변경합니다.
11. 승인완료 시 업체 잔액을 반영합니다.

## 5. 업체 잔액 출금 플로우

세움비즈의 출금은 사이트 내 자동 송금이 아닙니다.
사이트는 출금 요청과 상태 관리를 담당하고, 실제 계좌 확인 및 이체는 카카오톡과 운영자 수동 처리로 진행합니다.

업체 플로우:

1. 업체가 업체 잔액 내역을 확인합니다.
2. 업체가 업체 잔액 출금 메뉴로 이동합니다.
3. 출금 신청 금액을 입력합니다.
4. 사이트에서 출금 신청을 생성합니다.
5. 상태는 출금대기가 됩니다.
6. 출금 신청 시점에는 업체 잔액을 차감하지 않습니다.
7. 업체는 카카오톡에서 계좌번호를 전달합니다.

운영 원칙:

- 사이트 안에는 업체 계좌번호를 저장하지 않습니다.
- 자동 송금 기능은 없습니다.
- 실제 이체는 운영자가 수동 처리합니다.
- 출금 신청만으로는 업체 잔액 변동 로그를 만들지 않습니다.

## 6. 관리자 확인/지급승인 플로우

관리자 플로우:

1. 관리자가 출금대기 요청을 확인합니다.
2. 관리자가 카카오톡에서 업체 계좌 정보를 확인합니다.
3. 필요 시 상태를 지급진행중으로 변경합니다.
4. 운영자가 수동 이체를 진행합니다.
5. 관리자 페이지에서 지급승인 버튼을 클릭합니다.
6. 지급승인 시점에 업체 잔액을 차감합니다.
7. 상태를 지급완료로 변경합니다.
8. `biz_balance_ledger`에 차감 로그를 기록합니다.

반려/취소 플로우:

- 관리자가 반려하면 상태는 반려가 됩니다.
- 업체 또는 관리자가 취소하면 상태는 취소가 됩니다.
- 반려/취소 시 업체 잔액 차감은 없습니다.
- 반려/취소 시 업체 잔액 차감 로그도 생성하지 않습니다.

출금 상태값:

- 출금대기
- 지급진행중
- 지급완료
- 반려
- 취소

## 7. 이미지 핀번호 추출 플로우

목표:

- 상품권 이미지 업로드 후 핀번호 후보를 자동 추출합니다.
- OCR 또는 이미지 분석 기반으로 구현을 검토합니다.
- 상품권별 핀번호 패턴을 기준으로 후보를 정리합니다.
- 자동 추출 결과는 즉시 저장하지 않습니다.
- 업체가 확인/수정 후 최종 등록해야 저장됩니다.

추출 기준:

- 16자리 핀번호 패턴
- 18자리 핀번호 패턴
- 상품권별 구분자 포함 패턴
- 공백, 하이픈, 줄바꿈 제거 후 정규화

추후 검토:

- 금액 추출 가능성
- 상품권 종류 자동 판별 가능성
- 이미지 품질 낮음, 일부 가림, 회전 이미지 처리
- 중복 핀번호 감지
- OCR 실패 시 직접 입력 안내

## 8. 용어 기준

사용하지 않을 표현:

- 포인트
- 포인트 내역
- 포인트 적립
- 포인트 차감

사용할 표현:

- 업체 잔액
- 업체 잔액 내역
- 업체 잔액 반영
- 업체 잔액 차감

## 9. 구현 단계 우선순위

### 1단계: 문서와 화면 구조 정리

- 포털 기획 문서 확정
- DB 설계 문서 정리
- 인증/권한 설계 문서 정리
- 좌측 메뉴 기반 화면 설계

### 2단계: 업체 포털 화면 목업

- 공통 레이아웃
- 좌측 메뉴
- 매입 신청 화면
- 매입 내역 화면
- 업체 잔액 내역 화면
- 업체 잔액 출금 화면
- 내 정보 관리 화면

### 3단계: 데이터 구조 준비

- Supabase SQL 검토
- RLS 정책 검토
- 관리자 감사 로그 검토
- 이미지 업로드 보관 정책 검토

### 4단계: 기능 연결

- 로그인 방식 확정
- 매입 신청 저장
- 이미지 업로드
- OCR 또는 이미지 분석 연결
- 관리자 승인/반려 처리
- 업체 잔액 반영 로그 생성

### 5단계: 출금 운영 연결

- 업체 잔액 출금 신청
- 카카오톡 계좌 전달 안내
- 관리자 지급진행중/지급승인/반려/취소 처리
- 지급승인 시 업체 잔액 차감 로그 생성
